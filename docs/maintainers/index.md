# Maintainer Guide

This section is not meant for normal readers first. It exists so the project can grow without becoming chaotic.

A large public doctrine site needs maintenance rules. Without them, every new idea becomes a page, every page becomes stale, and the workbook/manual/archive split collapses.

## Maintainer goals

- preserve doctrine depth
- avoid unsupported meta claims
- keep page status visible
- separate public teaching from workbook operation
- keep private VOD critique anonymous
- verify patch-sensitive mechanics
- make the site readable for new players and useful for officers

## Current project layers

| Layer | Job |
|---|---|
| Website | Public hub and navigation |
| Manual pages | Teaching, role doctrine, fight concepts |
| Workbook packages | Live operating tools |
| Archive | Source preservation and project decisions |

## What maintainers should not do

- do not turn the public site into a random build dump
- do not delete useful doctrine just to shorten pages
- do not publish private guild critique with names
- do not add future comp packages without reliable source support
- do not hardcode patch-sensitive values unless recently verified
