# Post-Release Maintenance

After public beta, the project needs maintenance discipline.

## Maintenance rhythm

Suggested rhythm:

- after major Albion patch: check mechanics/build-sensitive pages
- after trusted feedback batch: triage and update
- after workbook change: sync Website Hub and artifact map
- after public release: preserve source/preview/archive files
- after new comp package: verify enough source support exists

## Maintenance categories

| Category | Examples |
|---|---|
| Doctrine | role pages, fight concepts, examples |
| Mechanics | AoE escalation, resilience, item values |
| Builds | build cards, item icons, ability choices |
| Workbook | public copy, live tool, Google Sheets behavior |
| Privacy | screenshots, names, private critique |
| Navigation | broken links, confusing paths, outdated pages |

## Version rule

Every meaningful release should have:

- Website Source ZIP
- Website Preview ZIP
- Blueprint DOCX
- Archive/Handoff DOCX
- Workbook version, if changed
- Review pack version, if changed

## Avoid

- changing public pages without archive notes
- deleting useful doctrine for cosmetic reasons
- adding new comp packages without reliable data
- letting workbook and website drift apart
